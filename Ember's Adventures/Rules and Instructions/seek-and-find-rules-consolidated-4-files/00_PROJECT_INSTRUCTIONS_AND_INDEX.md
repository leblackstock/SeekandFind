## System Instruction

Use these four uploaded files as the governing source for planning, prompting, reviewing, and revising seek-and-find books. Apply universal rules first, then audience overlays, mode overlays, project/theme overlays, workflows, QA, templates, and prompt blocks. Do not create copyrighted character clones or Waldo-like trade dress. For every page, preserve fair hiding, strong readability, print readiness, and commercial appeal. When unsure, ask one concise clarifying question or make a clearly labeled best-effort assumption.

Hard rule for seek-image puzzle art: never mark the mission item or any hidden object with a circle, square, box, outline, arrow, halo, label, highlight, or other answer-mark/shape overlay. The object must be hidden as natural scene art. Markers are allowed only in separate answer-key or hint assets created after final art approval, never on the seek page image itself.

For Ember books, production checklists are not source of truth. The source of truth is this four-file rules bundle plus `CHATGPT_PROJECT_SYSTEM_INSTRUCTIONS.md`; use the canonical Ember project rules, character canon, book plans, story/list source blocks, cover/KDP notes, and preflight rules now embedded inside these four files. # Seek-and-Find Rules System — Start Here

This rules system governs the planning, prompting, review, and production workflow for seek-and-find books across many themes, audiences, and formats.

It is designed to support:
- children's seek-and-find books
- adult seek-and-find books
- mascot-based series
- non-mascot hidden-object books
- seek-and-find coloring books
- future themed overlays such as fantasy, ocean, holiday, spooky, farm, cozy, sci-fi, history, and more

## Core Principle

Every book must be:
1. clear in audience and product promise
2. visually appealing at both thumbnail and full-page size
3. fair to solve
4. replayable
5. legally distinct from existing famous properties
6. consistent enough to feel professional
7. practical for print production

## How to Use This System

For each new book or series:

1. Read this file's rule precedence section.
2. Select the correct audience overlay from `01_RULES_CORE_AUDIENCE_MODE_PROJECTS.md`.
3. Select the correct mode overlay from `01_RULES_CORE_AUDIENCE_MODE_PROJECTS.md`.
4. Create or update a project/theme overlay using `01_RULES_CORE_AUDIENCE_MODE_PROJECTS.md` and `03_TEMPLATES_PROMPTS_AND_LOGS.md`.
5. Use templates from `03_TEMPLATES_PROMPTS_AND_LOGS.md` to define the series, book, page, and hidden-object list.
6. Build image prompts using `02_WORKFLOWS_AND_QA.md` and prompt blocks from `03_TEMPLATES_PROMPTS_AND_LOGS.md`.
7. Review every page using the QA checklists in `02_WORKFLOWS_AND_QA.md`.
8. Record major decisions in the decision log section of `03_TEMPLATES_PROMPTS_AND_LOGS.md`.

## What Belongs Where

- Universal rules, audience rules, mode rules, and project/theme overlays go in `01_RULES_CORE_AUDIENCE_MODE_PROJECTS.md`.
- Step-by-step processes and pass/fail QA checks go in `02_WORKFLOWS_AND_QA.md`.
- Reusable project documents, prompt language, prompt skeletons, and logs go in `03_TEMPLATES_PROMPTS_AND_LOGS.md`.
- Operating instructions, precedence, glossary, and model-use notes go in `00_PROJECT_INSTRUCTIONS_AND_INDEX.md`.

## Rule Writing Standard

Rules should be:
- specific
- testable
- reusable
- not tied to one theme unless placed in a project/theme overlay
- written to prevent common failure, not to decorate the page

Bad rule:
> Make the page nice.

Good rule:
> The page must have clear foreground, midground, and background zones so hidden objects can be distributed without visual mud.

## Production Warning

Do not build a full book from unreviewed generated images. Every page must pass page QA before being included in a book project. A pretty failure is still a failure, just with better lighting.

# Rule Precedence

This file defines which rules win when two files appear to conflict.

## Precedence Order

Rules apply in this order:

1. **Core universal rules** — `01_RULES_CORE_AUDIENCE_MODE_PROJECTS.md`
2. **Audience rules** — `01_RULES_CORE_AUDIENCE_MODE_PROJECTS.md`
3. **Mode rules** — `01_RULES_CORE_AUDIENCE_MODE_PROJECTS.md`
4. **Project/theme rules** — `01_RULES_CORE_AUDIENCE_MODE_PROJECTS.md`
5. **Workflow rules** — `02_WORKFLOWS_AND_QA.md`
6. **QA checklists** — `02_WORKFLOWS_AND_QA.md`
7. **Templates and prompt blocks** — `03_TEMPLATES_PROMPTS_AND_LOGS.md`

## Override Policy

Lower-precedence files may add constraints, but they may not silently override higher-precedence rules.

Example:
- Core rule: no copyrighted character imitation.
- Theme overlay cannot override this by asking for a character who resembles a protected property.

## Project Overlay Policy

A project/theme overlay may define:
- theme vocabulary
- mascot design
- page list
- recurring visual motifs
- object categories
- commercial angle
- audience selection
- mode selection
- difficulty target

A project/theme overlay may not:
- weaken legal safety rules
- remove print-safety requirements
- ignore audience age suitability
- permit unfair hiding
- permit unreadable art

## Conflict Resolution

When conflict occurs:
1. preserve legal and safety rules first
2. preserve audience suitability second
3. preserve print readiness third
4. preserve gameplay fairness fourth
5. preserve theme flavor last

## Documentation Requirement

Any intentional exception must be recorded in:
- the relevant project overlay
- the decision log section of `03_TEMPLATES_PROMPTS_AND_LOGS.md`

Exception entries must include:
- date
- rule affected
- reason
- risk
- approval/status

## Default Rule

If uncertain, choose the rule that creates a clearer, safer, more commercially usable book.

# Glossary

## Seek-and-Find

A puzzle-page format where the viewer searches a detailed illustration for one or more hidden characters or objects.

## Mascot

A recurring character that appears across pages or books. Mascots create brand recognition and series identity.

## Hidden Object

Any item the viewer is expected to locate within the illustration.

## Primary Target

The most important thing to find, usually the mascot or main hidden character.

## Secondary Target

A listed hidden object that supports the puzzle but is not the main recurring character.

## Fair Hiding

A hiding method where the object is challenging but identifiable once noticed.

## Unfair Hiding

A hiding method where the object is too tiny, too distorted, too obscured, too low-contrast, placed on the extreme edge, or visually impossible to identify.

## Visual Hierarchy

The organization of focal points, detail density, contrast, and scale so the page is readable rather than chaotic.

## Detail Density

The amount of visual information in a page or zone. High detail density can be good, but only when organized.

## Visual Mud

Overcrowding, low contrast, unclear shapes, or over-rendering that makes the page hard to read or unpleasant to solve.

## Difficulty Curve

The planned progression of challenge across a book or series.

## Thumbnail Appeal

How attractive and understandable the book appears at small Amazon thumbnail size.

## Print Safety

Design choices that prevent important details from being lost due to trim, binding, scaling, paper quality, or print contrast.

## Seek-and-Color

A hybrid format where the page functions as both a hidden-object puzzle and a coloring page.

## Overlay

A rule file that adds constraints for a specific audience, mode, theme, or project without replacing the core rules.

# Model Use Notes

This system can be used with image-generation models, writing models, layout tools, and review agents.

## Model Roles

Recommended division of labor:

1. **Planner model**
   - defines concept
   - creates page briefs
   - checks rule compliance
   - builds prompts

2. **Image model**
   - generates the illustration

3. **Reviewer model**
   - evaluates fairness, readability, consistency, and print risk

4. **Layout/book production tool**
   - assembles pages into the final print-ready book

## Image Prompt Rule

Image prompts must be built from:
- book brief
- page brief
- audience overlay
- mode overlay
- project/theme overlay
- universal hiding rules
- print requirements

Do not prompt from vibes alone.

For title pages in mascot books, the prompt must reference the character images saved in ChatGPT Sources by their exact uploaded names and must include the written mascot description from the active project canon. For Ember books, the title-page prompt must directly name `Ember-001`, `Ember-002`, and `Ember-003`. Title text must be the exact approved title words only, with no flexible wording, substitutions, extra words, fake text, or misspellings. Do not send prompts with unresolved placeholders or generic book labels. If the exact title, required source images, or written mascot description are missing, stop and ask instead of generating.

## Image Review Rule

Generated images are not automatically approved. A generated image must pass:
- page QA
- hiding QA
- audience QA
- print-risk review
- legal-safety review

## Consistency Rule

For mascot books, every prompt must repeat the mascot design details unless a reliable reference image/control system is being used.

## Revision Rule

When revising a generated image, identify the specific defect.

Bad revision:
> Make it better.

Good revision:
> Move the mascot away from the page edge, increase contrast on the teal scarf, and redistribute hidden objects so no more than three are in the foreground.

## Model Limitation Rule

Image models often fail at exact counts, tiny object placement, and consistent character design. Treat generated outputs as drafts until manually reviewed.
